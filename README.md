# rain-research-bot

> IT 개념 질문과 최신 시사·기업 뉴스를 구분해서, 알맞은 도구로 조사해 답하는 LangGraph 기반 리서치 에이전트

`Python` `LangChain` `LangGraph` `FastAPI`

---

## 소개

"OAuth가 뭐야?" 같은 개념 질문과 "오늘 삼성전자 뉴스 있어?" 같은 시사 질문은 정보의 성격이 다르다. 전자는 시간이 지나도 안 바뀌는 지식이고, 후자는 매번 바뀌는 정보다. 이 프로젝트는 이 둘을 하나의 파이프라인으로 뭉뚱그리지 않고, Agent가 질문 성격을 스스로 판단해서 알맞은 도구를 고르도록 설계했다.

```
IT 개념/배경지식 질문 -> 미리 구축해둔 위키 기반 인덱스에서 검색 (정적, 빠름)
실시간 시사/기업 정보 -> 네이버 뉴스 API로 그때그때 검색 (동적, 최신)
```

---

## 주요 기능

- **IT 개념 검색**: 한국어 위키백과 기반 하이브리드 검색(FAISS 의미 검색 + BM25 키워드 검색)
- **실시간 뉴스 검색**: 네이버 뉴스 검색 API로 최신순 뉴스 조회
- **Agent 라우팅**: 두 도구 중 무엇을 쓸지 사람이 규칙을 짜지 않고, LLM이 tool-calling으로 스스로 판단
- **대화 기억**: 같은 세션(thread) 안에서는 이전 질문을 기억하고 이어서 답변
- **REST API**: FastAPI로 배포, Swagger UI에서 바로 테스트 가능

## RAG 아키텍처

**데이터 파이프라인 (사전 준비, 1회 실행)**

```
IT 시드 용어 30개
  └─ 한국어 위키백과에서 각 문서 수집
       └─ 문서 내 링크를 따라가며 확장 수집 (문서당 최대 15개)
            └─ 목록/동음이의/짧은 문서 필터링           -> 258개 문서 확보
                 └─ 부록 섹션(같이 보기·각주·외부 링크 등) 제거
                      └─ 500자 단위로 청킹 (겹침 50자, 청크 앞에 문서 제목 삽입)
                           └─ BAAI/bge-m3로 임베딩
                                └─ FAISS 인덱스로 저장            -> 청크 4,701개
```

**검색 단계 (질문마다 실행)**

```
질문
  ├─ FAISS 의미 검색  (유사도 0.5 이상만 채택)
  ├─ BM25 키워드 검색 (Mecab 명사 추출, 점수 12.0 이상만 채택)
  └─ 두 점수를 절반씩(alpha=0.5) 합쳐서 상위 3개 선정
       └─ 결과 0개면 -> 기준을 완화(0.3 / 6.0)해서 한 번 더 검색
            └─ 그래도 0개면 "관련 문서 없음"으로 정직하게 응답
```

## LangGraph 아키텍처

```
START
  └─ agent 노드 (LLM이 다음 행동 판단)
       ├─ 도구 호출이 필요 없으면 -> END (최종 답변)
       └─ 도구 호출이 필요하면
            └─ tools 노드 (실제 함수 실행)
                 ├─ search_it_wiki      -> IT 위키 하이브리드 인덱스
                 └─ search_realtime_news -> 네이버 뉴스 API
                      └─ 결과를 들고 agent 노드로 복귀 (반복)
```

`agent`와 `tools` 두 노드가, 더 이상 도구가 필요 없다고 판단할 때까지 서로를 오간다(ReAct 패턴). 이 판단은 `tools_condition`이 맡고, 실제 함수 실행은 `ToolNode`가 맡는다. 여기에 `MemorySaver`를 붙여, `thread_id`가 같은 요청끼리는 이전 대화 내용을 이어서 참조할 수 있게 했다.

## 요청 처리 흐름 (예시로 따라가기)

**예시 1 - IT 개념 질문**

```
사용자: "인공지능이 뭐야?"
  1) agent 노드: 질문을 보고 "배경지식 질문이네" 판단, search_it_wiki 호출 요청
  2) tools 노드: search_it_wiki("인공지능") 실행 -> 하이브리드 검색 결과 반환
  3) agent 노드: 검색 결과를 읽고 최종 답변 작성 -> 도구 더 필요 없음 -> END
  4) 사용자에게 답변 + 어떤 도구를 썼는지(tool_used) 함께 반환
```

**예시 2 - 실시간 시사 질문**

```
사용자: "오늘 삼성전자 관련 뉴스 알려줘"
  1) agent 노드: "최신 정보가 필요한 질문이네" 판단, search_realtime_news 호출 요청
  2) tools 노드: 네이버 뉴스 API 호출 -> 최신순 뉴스 5건 반환
  3) agent 노드: 뉴스들을 요약해서 최종 답변 작성 -> END
```

**예시 3 - 이어지는 질문 (대화 기억)**

```
사용자(thread_id="room1"): "인공지능이 뭐야?"  -> 답변 A
사용자(thread_id="room1"): "방금 내가 뭐라고 물어봤어?"
  -> MemorySaver가 같은 thread_id의 이전 messages를 불러옴
  -> agent 노드가 이전 질문 내용을 참조해서 정확히 답변
```

## 기술 스택

| 영역 | 선택 | 이유 |
|---|---|---|
| 오케스트레이션 | LangGraph (`StateGraph(MessagesState)`) | 조건 분기·재시도를 표현하려면 일직선 파이프로는 부족 |
| 도구 판단 | tool-calling (`bind_tools` + `ToolNode` + `tools_condition`) | 키워드 매칭이 아니라 LLM이 도구 설명(docstring)을 보고 직접 판단 |
| 대화 기억 | `MemorySaver` + `thread_id` | 세션별로 대화 맥락을 유지 |
| IT 지식 소스 | 한국어 위키백과 (시드 + 링크 확장 수집) | IT 개념 설명에 특화된 문서를 직접 확보 |
| 검색 | 하이브리드 (FAISS + Mecab/BM25) | 의미 기반 검색과 키워드 기반 검색을 절반씩 결합, 관련 없는 질문은 자동 필터링 |
| 실시간 정보 | 네이버 뉴스 검색 API | 한국 기업 뉴스 품질이 좋고 무료 할당량이 넉넉함 |
| 생성 LLM | Gemini 2.5 Flash | 라우팅 판단과 답변 생성 모두 동일 모델로 처리 |
| 배포 | FastAPI + Uvicorn | REST API화, Swagger로 즉시 테스트 가능 |

## 트러블슈팅

**KorQuAD는 IT 데이터가 아니었다**
기존에 재사용하려던 KorQuAD는 위키백과 전반(역사·인물·문화)에서 뽑은 일반 데이터셋이었다. IT 개념 질문에는 맞지 않는다는 걸 확인하고, IT 전용 위키 문서를 시드+링크 확장 방식으로 직접 수집했다.

**위키 문서 수집 라이브러리 버전 충돌**
`wikipedia-api` 최신 버전이 최신 파이썬 문법(`|` 타입 유니온)을 써서 구버전 파이썬 환경에서 `TypeError`가 발생했다. 호환되는 버전으로 고정해서 해결했다.

**청크에 위키 부록 섹션이 섞여 들어감**
검색 테스트 중 소프트웨어 라이브러리 목록이나 "같이 보기/각주/외부 링크" 같은 부록 섹션이 그대로 청크로 잡히는 걸 발견했다. 위키 재수집 없이, 저장된 텍스트에서 부록 섹션 제목이 나오는 지점부터 잘라내는 후처리로 해결했다(문서당 평균 252자 제거).

**threshold 경계선에 걸린 검색 실패**
IT 인덱스에 분명히 있는 "인공지능" 문서조차 0건으로 검색되는 경우가 있었다. 원인은 threshold 경계선 — 검색 기준(0.5 / 12.0)에 근소하게 못 미치는 경우였다. 1차 검색이 0건이면 완화된 기준(0.3 / 6.0)으로 2차 검색을 자동 시도하도록 개선했다. 임베딩/BM25 인덱스를 다시 만들지 않고, 이미 계산된 점수를 다른 기준으로 재평가하는 방식이라 속도 부담이 없다.

**뉴스 원문 크롤링은 링크마다 접근 가능 여부가 다름**
네이버 뉴스 API는 제목+짧은 요약만 제공한다. 기사 원문 전체를 읽으려면 `link`를 열어야 하는데, 언론사 자체 사이트는 접근이 가능했지만 네이버가 직접 호스팅하는 뉴스 페이지는 접근이 막혀 있었다. 언론사별 예외 처리가 필요해 지금은 요약 정보만 쓰고, 원문 크롤링은 스트레치골로 남겼다.

**검색 결과가 부족해도 정직하게 보완함**
IT 위키 인덱스에 없는 개념(tool-calling 등 최신 AI 용어)을 물었을 때, Agent는 "검색 결과에 직접적인 설명이 부족하다"는 걸 답변에 명시한 뒤, 자체 지식으로 개념을 보완해서 답했다. 검색이 못 찾아도 생성 단계가 정직하게 보완하는 특징을 확인했다.

## 스트레치 골

1. 뉴스 원문 크롤링 - 언론사 자체 사이트는 원문 전체를 가져와 청킹·임베딩까지 하고, 네이버 자체 페이지는 접근 제한 우회 방안 검토
2. MiniGPT Stage 2 파인튜닝 후, 동일한 평가 지표로 MiniGPT vs Gemini 비교 평가
3. Query Expansion / Reranking으로 동음이의어 오탐 개선
4. 다양한 문서 형식 로더(md/html/json/pdf 자동 분기) 추가
5. 대화 기억을 메모리 대신 디스크에 영구 저장(SqliteSaver 등)
